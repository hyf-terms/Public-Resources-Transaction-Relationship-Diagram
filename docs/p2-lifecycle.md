# P2 更正、终止与重新采购生命周期

P2 在 P0 事件库和 P1 三阶段链路上增加生命周期关联，目标是区分“公告事件”和“经济需求状态”，避免把更正、废标、二次招标分别计算成多笔订单。

## 支持的生命周期关系

| 公告事件 | P2关系 | 受影响对象 |
|---|---|---|
| 普通更正 | `AMENDS` | 采购轮次 `attempt` |
| 开标或截止日期延期 | `EXTENDS_DEADLINE` | 采购轮次 `attempt` |
| 废标、流标、采购失败 | `FAILS` | 标包 `package` |
| 项目取消或终止 | `TERMINATES` | 采购轮次 `attempt` |
| 二次、重新招标或重新采购 | `RETRY_OF` | 前一采购轮次→新采购轮次 |

更正和延期不直接改变项目的当前状态；废标只影响对应标包；重新采购创建新轮次，并保留它与原轮次的关系。

## 新增表

- `lifecycle_document_feature`：各文档版本的生命周期匹配特征；
- `lifecycle_link_candidate`：每个事件最多三个前序公告候选；
- `lifecycle_link`：更正、终止或重新采购与前序公告及业务对象的关联；
- `lifecycle_review_queue`：无候选、低分、歧义或无法定位业务对象的复核任务；
- `current_lifecycle_link`：每个事件当前得分最高的生命周期关联视图。

P2向同一个 `procurement.sqlite` 追加版本3结构，P0和P1数据不删除、不覆盖。

## 匹配规则

匹配只允许从更早的招标/中标公告指向更晚的生命周期事件：

```text
招标/中标 → 更正或延期
招标/中标 → 废标或终止
第一轮招标 → 第二轮或重新采购
```

硬性排除规则：

1. 项目编号两侧均存在但不一致；
2. 标包号两侧均存在但不一致；
3. 前序公告晚于事件公告；
4. 公告间隔超过730天；
5. `RETRY_OF` 的新轮次不高于原轮次。

自动接受仍要求项目编号完全一致、分数不低于0.85且前两名候选分差不小于0.05。其他情况进入人工复核，不强行关联。

## 新增输出

- `lifecycle_links.csv`：当前生命周期关联、受影响对象、匹配分数和证据；
- `lifecycle_review_queue.csv`：P2待人工复核任务；
- `object_lifecycle_state.csv`：采购轮次和标包的当前状态；
- `data_quality_metrics.csv`：文档、版本、P1/P2链路及待复核数量等质量指标。

`object_lifecycle_state.csv` 的状态来自 P0 追加式事件视图；`lifecycle_links.csv` 解释更正、终止和重采公告实际关联到哪个轮次或标包。

## 数据质量口径

当前质量文件包括：

- 来源逻辑文档数；
- 不可变文档版本数；
- 出现多个网页版本的文档数；
- P1当前链路数和待复核数；
- P2当前生命周期链路数和待复核数；
- 各标准公告阶段文档数量。

它是每次运行后的数据快照，不等同于自动定时监控。抓取成功率、站点中断和发布时间—首次发现延迟需要后续结合运行日志继续完善。

## 重要限制

- 当前只解析正文中的基础项目编号和标包号，复杂附件表格仍可能进入复核；
- 标题缺少项目编号时不会自动接受；
- 如果前序招标从未被抓取，终止公告会标为 `NO_CANDIDATE`；
- P2不把失败后的重新采购金额与原失败标包金额相加；
- 尚未加入采购意向和验收阶段。
